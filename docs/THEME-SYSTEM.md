# Theme System

Cross-tool dark/light theming across ghostty, lazygit, tmux, yazi, btop, fzf,
nvim, and Claude Code, driven by macOS appearance.

## When something looks wrong

Run `theme`. That's the whole answer.

| Command | What it does |
|---|---|
| `theme` | Re-apply current macOS appearance to every CLI tool |
| `theme-dark` | Flip macOS to dark + sync all tools |
| `theme-light` | Flip macOS to light + sync all tools |

If `theme` doesn't fix it, the daemon is probably stuck ÔÇö restart it:

```bash
pkill -f "bin/appearance-watcher" && nohup ~/.dotfiles/bin/appearance-watcher >/dev/null 2>&1 & disown
```

## Architecture

```
macOS appearance flip
        Ôćô
appearance-watcher (daemon, polls every 2s)
        Ôćô
        ÔöťÔöÇ btop:    sed btop.conf Ôćĺ vulpes-reddish2-{mode}-btop
        ÔöťÔöÇ lazygit: cp vulpes-reddishnovember-{mode}.yml Ôćĺ config.yml
        ÔöťÔöÇ tmux:    sed .tmux.conf source-file line + tmux source-file
        ÔöťÔöÇ yazi:    ln -sf vulpes-reddishnovember-{mode}.toml Ôćĺ theme.toml
        ÔöťÔöÇ fzf:     ln -sf theme-{mode}.sh Ôćĺ current.sh
        ÔööÔöÇ claude:  jq .theme = "{mode}" in ~/.claude.json
```

Daemon: `~/.dotfiles/bin/appearance-watcher` (launched from `.startup.sh`,
self-restarted via `pgrep -qf` guard). Only writes on mode *change* ÔÇö file
state diverging from macOS state without a flip won't self-heal. That's why
`theme --sync` exists.

Nvim handles its own flip via `auto-dark-mode.nvim` (in
`~/.dotfiles/.config/nvim/lua/plugins/theming.lua`); the watcher doesn't touch
nvim.

Session-local colors (zsh prompt palette + `FZF_DEFAULT_OPTS`) are the one leg
a daemon can't do ÔÇö `.zshrc` sources `.config/zsh/themes/ÔÇŽ-{mode}.zsh` and
`~/.config/fzf/current.sh` at shell start. That block is read-only: the old
`vulpes-dark`/`vulpes-light` zshrc functions that rewrote lazygit config via
heredoc were removed 2026-07-13 (they fought the watcher); they survive as
aliases to `theme-dark`/`theme-light`/`theme`.

## Source-of-truth files per tool

| Tool | Source file(s) | Active config |
|---|---|---|
| ghostty | `.config/ghostty/themes/vulpes-reddishnovember-{dark,light}` | (no flip ÔÇö ghostty reads OS appearance natively via `theme = light:ÔÇŽ,dark:ÔÇŽ` config line) |
| lazygit | `.config/lazygit/vulpes-reddishnovember-{dark,light}.yml` | `.config/lazygit/config.yml` (gitignored ÔÇö auto-rewritten) |
| tmux | `vulpes-reddish2-{dark,light}-tmux.tmux.conf` (in dotfiles root) | `.tmux.conf` source-file line (skip-worktree) |
| yazi | `.config/yazi/vulpes-reddishnovember-{dark,light}.toml` | `.config/yazi/theme.toml` symlink (gitignored) |
| btop | `.config/btop/themes/vulpes-reddish2-{dark,light}-btop.theme` | `.config/btop/btop.conf` (skip-worktree) |
| fzf | `.config/fzf/theme-{dark,light}.sh` | `.config/fzf/current.sh` symlink |
| nvim | `.config/nvim/colors/vulpes-reddishnovember-{dark,light}.lua` | (no live config ÔÇö auto-dark-mode runs `:colorscheme`) |

Canonical palette is `~/code/VULPES/vulpes.nvim/lua/vulpes/palette.lua`. Per-tool
exports are generated by `vulpes-theme-lab`, but most have drifted via
hand-edits and AI tweaks ÔÇö there's no regeneration pipeline.

## Why some files are skip-worktree / gitignored

The watcher mutates active config files in place every 2s on mode flips.
Originally those active files (`yazi/theme.toml`, `lazygit/config.yml`,
`btop.conf`, `.tmux.conf`) were tracked, leaving the dotfiles repo
permanently dirty against whichever mode was current at last commit.

Fix applied 2026-04-28 (commit `88bfbef`):

- `yazi/theme.toml` and `lazygit/config.yml` Ôćĺ **untracked + gitignored**
  (their content is fully derivable from sibling `*-{dark,light}.{toml,yml}`
  files which ARE tracked).
- `btop.conf` and `.tmux.conf` Ôćĺ **still tracked, marked `skip-worktree`**
  (they're real configs where only 1ÔÇô2 lines flip; the rest is worth
  versioning).

Fresh clone needs:

```bash
git update-index --skip-worktree .config/btop/btop.conf .tmux.conf
```

## Pitfall: `git checkout` desyncs everything

Running `git checkout -- <file>` on any of the auto-flipped files restores
the index version (whichever mode was current when last committed) and the
watcher won't re-correct because no mode *change* occurred. Always run
`theme` after a checkout that touches these files.

## Known cruft (someday-maybe)

- **`vulpes-reddish2` references in tmux + btop.** The rest of the ecosystem
  migrated to `vulpes-reddishnovember` in November 2025, but tmux and btop
  still reference `reddish2`. The `reddish2` source files exist and work,
  but for consistency they should be regenerated under the
  `reddishnovember` name. Needs `vulpes-theme-lab` work, not just renames.
- **Yesterday's v2.5 commit (`41587c2`) made unverified contrast claims**
  (all light pairs Ôëą4.5:1, HCL-harmonized diffs). Worth a chroma-js audit
  before fully trusting.
- **`appearance-watcher` doesn't survive across launchd reboots cleanly**
  ÔÇö relies on `.startup.sh` running on shell launch. A LaunchAgent plist
  would be more robust.

## How we got here (short version)

October 2025: vulpes theme system established, multiple rapid renames
(`reddish` Ôćĺ `reddish2` Ôćĺ `reddishnovember`) leaving partial migrations.
April 2026: `light-vulpes-v2.5` redesign + new appearance-watcher daemon
landed in two consecutive commits, but the watcher's writes hit tracked
files and turned every mode flip into dirty git state. AI-coauthored
"diversify palette" commit drifted ANSI colors away from canonical vulpes
pink. Cleanup commits 2026-04-28 (`1e1e4f2`, `88bfbef`) and 2026-04-29
(`4947ba5`, `4c05509`, `cd8adf2`) restored the system to a sane baseline:
canonical palette back, dirty state eliminated, manual `theme` rescue
trio added.
